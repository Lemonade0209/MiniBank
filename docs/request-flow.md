# MiniBank 요청 흐름

## 1. 문서 목적

이 문서는 사용자의 화면 조작이 Controller, Service, Repository를 거쳐 어떤 화면으로 이어지는지 정리한다.

특히 회원가입, 계좌 개설, 입금, 출금, 계좌이체처럼 데이터를 바꾸는 요청은 **POST 처리 후 GET 화면으로 redirect**한다. 이를 통해 브라우저 새로고침으로 같은 요청이 다시 제출되는 문제를 줄인다.

URL, Method와 상태 코드는 `http-api.md`, 기능과 업무 규칙은 `requirements.md`를 기준으로 한다. 이 문서에서는 인증, 정상·실패 흐름과 redirect 목적지를 다룬다.

## 2. 공통 처리 원칙

### 2.1 계층별 역할

```text
브라우저
  -> Controller: 요청값 수신, 입력 형식 검증
  -> Service: 로그인 회원, 소유권, 잔액, 거래 규칙 검증
  -> Repository: 데이터 조회와 저장
  -> Controller: 화면 반환 또는 redirect
```

- 입력값의 누락, 길이, 숫자 형식처럼 폼에 가까운 검증은 Controller와 폼 DTO에서 처리한다.
- 계좌 소유권, 잔액 부족, 동일 계좌 이체처럼 비즈니스 규칙에 해당하는 검증은 Service에서 처리한다.
- 데이터 변경과 거래내역 저장은 Service가 하나의 작업으로 묶는다.

### 2.2 GET과 POST

| 요청 | 용도 | 처리 결과 |
|---|---|---|
| GET | 화면과 데이터 조회 | Thymeleaf 화면 반환 |
| POST | 회원·세션·계좌·잔액 변경 | 성공 시 3xx redirect로 GET URL에 이동 |

상태를 바꾸는 기능은 GET으로 만들지 않는다.

### 2.3 성공과 실패

| 상황 | 처리 방식 |
|---|---|
| GET 성공 | 요청한 화면을 반환한다. |
| POST 성공 | 정해진 GET 화면으로 redirect한다. 필요하면 일회성 성공 메시지를 함께 전달한다. |
| 폼 입력 오류 | 저장하지 않고 오류와 입력값을 담아 같은 화면을 반환한다. |
| 비즈니스 규칙 위반 | 데이터를 바꾸지 않고 해당 화면에 실패 이유를 보여준다. |
| 비로그인 접근 | 로그인 화면으로 redirect한다. |
| 예상하지 못한 오류 | 일부 데이터가 남지 않도록 롤백하고 오류 처리 흐름으로 넘긴다. |

입력 오류 때 redirect하지 않는 이유는 검증 결과와 사용자가 입력한 값을 같은 요청 안에서 보여주기 위해서다. 비밀번호는 오류 화면에 다시 채우지 않는다.

### 2.4 PRG가 막아주는 것과 막지 못하는 것

PRG는 POST 성공 후 새로고침으로 같은 요청이 다시 전송되는 문제를 막는다. 다만 사용자가 응답 전에 버튼을 빠르게 두 번 누르는 상황까지 완전히 막지는 못한다.

- 화면에서는 제출 직후 버튼을 비활성화할 수 있다.
- 입금·출금·이체의 최종 안전성은 Service와 DB 트랜잭션이 보장한다.

## 3. 인증 공통 흐름

### 3.1 세션과 접근 경로

| 항목 | 설계 기준 |
|---|---|
| 세션 키 | `SessionConst.LOGIN_MEMBER_ID` |
| 세션 값 | 로그인한 회원의 `Long memberId` |
| 세션 쿠키 | Spring Boot의 기본 `JSESSIONID` 사용 예정 |
| 유휴 만료 | 마지막 요청 후 30분 |
| 로그인 성공 | 기존 세션을 폐기하고 새 세션 생성 |
| 로그아웃 | 현재 세션 전체 무효화 |

세션에는 회원 객체나 비밀번호를 넣지 않는다. `HttpOnly`, `SameSite`, `Secure` 같은 쿠키 속성은 MVC2에서 Spring Boot의 실제 설정과 응답을 확인한 뒤 확정한다. 운영 환경에서 HTTPS를 사용한다면 `Secure` 적용을 검토한다.

| 구분 | 경로 |
|---|---|
| 공개 | `/`, `/health`, `/members/add`, `/login` |
| 정적·오류 | `/css/**`, `/js/**`, `/images/**`, `/favicon.ico`, `/error` |
| 보호 | `/accounts/**`, `/transfers/**`, `/logout` |

`LoginCheckInterceptor`는 로그인 여부와 복귀 URL 생성까지만 담당한다. 회원 조회, 계좌 소유권과 잔액 검증은 Service에서 처리한다.

### 3.2 보호된 GET 화면 접근

```text
1. 비로그인 사용자가 보호된 GET URL 요청
2. 인터셉터가 로그인 여부 확인
3. /login?redirectURL={원래 GET 경로} 로 redirect
4. 로그인 화면 반환
5. 로그인 성공
6. 검증된 원래 GET 경로로 redirect
```

- `redirectURL`은 `/` 하나로 시작하는 애플리케이션 내부 GET 경로만 허용한다.
- `//`, `http://`, `https://`, 역슬래시와 줄바꿈 문자가 포함된 값은 거부한다.
- 허용 대상은 `/`, `/accounts`, 계좌 상세·거래내역과 이체 폼이다.
- 외부 URL이거나 유효하지 않으면 `/accounts`로 이동한다.
- 검증은 URL decoding 후 수행하고 로그인 URL에 넣을 때 다시 URL encoding한다.

### 3.3 보호된 POST 요청

```text
1. 비로그인 사용자가 보호된 POST URL 요청
2. 인터셉터가 요청 차단
3. 로그인 화면으로 redirect
4. 로그인 성공 후 /accounts 로 redirect
```

로그인 전에 차단된 POST 요청은 자동으로 재실행하지 않는다. 거래 요청 본문을 임의로 보관하거나 재전송하면 중복 거래와 보안 문제가 생길 수 있기 때문이다.

## 4. 회원가입

### 4.1 정상 흐름

```text
GET  /members/add
  -> 회원가입 화면

POST /members/add
  -> 입력값 검증
  -> 로그인 ID 중복 확인
  -> 회원 저장
  -> redirect:/login

GET  /login
  -> 로그인 화면
```

| 단계 | 처리 내용 |
|---|---|
| 폼 조회 | 빈 회원가입 폼을 보여준다. |
| 입력 검증 | 로그인 ID, 비밀번호, 이름의 필수값과 형식을 확인한다. |
| 중복 확인 | 같은 로그인 ID가 있는지 확인한다. |
| 저장 | 검증을 통과한 회원을 저장한다. |
| redirect | 로그인 화면으로 이동한다. |

### 4.2 실패 흐름

- 필수값이나 형식이 잘못되면 회원을 저장하지 않고 회원가입 화면을 다시 보여준다.
- 로그인 ID가 중복되면 해당 필드에 오류를 표시한다.
- 사용자가 입력한 로그인 ID와 이름은 유지할 수 있지만 비밀번호는 다시 보여주지 않는다.

## 5. 로그인과 로그아웃

### 5.1 로그인

```text
GET  /login
  -> 로그인 화면

POST /login
  -> 로그인 ID와 비밀번호 확인
  -> 세션 생성
  -> redirect:{안전한 redirectURL 또는 /accounts}
```

- 로그인 정보가 틀리면 세션을 만들지 않고 로그인 화면에 오류를 보여준다.
- `redirectURL`이 없으면 `/accounts`로 이동한다.

### 5.2 로그아웃

```text
POST /logout
  -> 세션 제거
  -> redirect:/

GET  /
  -> 홈 화면
```

로그아웃은 세션을 변경하는 요청이므로 GET이 아니라 POST를 사용한다.

## 6. 계좌 개설

### 6.1 정상 흐름

```text
GET  /accounts/add
  -> 계좌 개설 확인 화면

POST /accounts/add
  -> 로그인 회원 확인
  -> 중복되지 않는 계좌번호 생성
  -> 잔액 0원 계좌 저장
  -> redirect:/accounts

GET  /accounts
  -> 새 계좌가 포함된 내 계좌 목록
```

- v1.0에서는 계좌 이름과 초기 잔액을 입력받지 않는다.
- 계좌번호는 서버에서 생성한다.

### 6.2 실패 흐름

- 세션이 없으면 로그인 화면으로 이동한다.
- 생성한 계좌번호가 이미 존재하면 저장하지 않고 새 번호를 만들어 다시 시도한다.
- 저장에 실패하면 계좌가 생성된 것처럼 성공 화면을 보여주지 않는다.

## 7. 입금

입금 폼은 계좌 상세 화면 안에 둔다.

### 7.1 정상 흐름

```text
GET  /accounts/{accountId}
  -> 계좌 상세와 입금 폼

POST /accounts/{accountId}/deposit
  -> 로그인 회원 확인
  -> 계좌 존재 여부와 소유권 확인
  -> 금액 검증
  -> 잔액 증가
  -> 입금 거래내역 저장
  -> redirect:/accounts/{accountId}

GET  /accounts/{accountId}
  -> 변경된 잔액과 성공 메시지
```

잔액 증가와 거래내역 저장은 함께 성공해야 한다.

### 7.2 실패 흐름

| 상황 | 결과 |
|---|---|
| 금액이 비어 있거나 숫자가 아님 | 계좌 상세 화면에 입력 오류 표시 |
| 금액이 0원 이하 | 잔액과 내역을 바꾸지 않고 오류 표시 |
| 존재하지 않는 계좌 | 데이터 변경 없이 계좌 오류 처리 |
| 다른 회원의 계좌 | 데이터 변경 없이 접근 거부 |
| 거래내역 저장 중 오류 | 잔액 증가까지 함께 롤백 |

입력 오류로 계좌 상세 화면을 다시 보여줄 때는 계좌번호와 현재 잔액을 다시 조회해 모델에 담는다.

## 8. 출금

출금 폼도 계좌 상세 화면 안에 둔다.

### 8.1 정상 흐름

```text
GET  /accounts/{accountId}
  -> 계좌 상세와 출금 폼

POST /accounts/{accountId}/withdraw
  -> 로그인 회원 확인
  -> 계좌 존재 여부와 소유권 확인
  -> 금액 검증
  -> 잔액이 충분할 때만 차감
  -> 출금 거래내역 저장
  -> redirect:/accounts/{accountId}

GET  /accounts/{accountId}
  -> 변경된 잔액과 성공 메시지
```

### 8.2 실패 흐름

| 상황 | 결과 |
|---|---|
| 금액 형식 오류 또는 0원 이하 | 잔액과 내역을 바꾸지 않고 입력 오류 표시 |
| 잔액 부족 | 잔액과 내역을 바꾸지 않고 실패 이유 표시 |
| 존재하지 않는 계좌 | 데이터 변경 없이 계좌 오류 처리 |
| 다른 회원의 계좌 | 데이터 변경 없이 접근 거부 |
| 거래내역 저장 중 오류 | 잔액 차감까지 함께 롤백 |
| 동시 출금으로 먼저 잔액이 사용됨 | 조건부 `UPDATE` 실패로 판단하고 내역을 만들지 않음 |

## 9. 계좌이체

### 9.1 정상 흐름

```text
GET  /transfers/new?fromAccountId={accountId}
  -> 출금 계좌가 선택된 이체 화면

POST /transfers/new
  -> 로그인 회원 확인
  -> 출금 계좌 존재 여부와 소유권 확인
  -> 입금 계좌번호 조회
  -> 동일 계좌 여부 확인
  -> 금액과 잔액 확인
  -> 출금 계좌 잔액 차감
  -> 입금 계좌 잔액 증가
  -> 같은 transferGroupId로 거래내역 2건 저장
  -> redirect:/accounts/{fromAccountId}

GET  /accounts/{fromAccountId}
  -> 변경된 잔액과 이체 성공 메시지
```

출금, 입금, 거래내역 2건 저장은 하나의 트랜잭션으로 처리한다.

### 9.2 실패 흐름

| 상황 | 결과 |
|---|---|
| 출금 계좌가 본인 소유가 아님 | 이체 거부, 데이터 변경 없음 |
| 입금 계좌번호가 존재하지 않음 | 이체 화면에 오류 표시 |
| 출금·입금 계좌가 같음 | 이체 화면에 오류 표시 |
| 금액 형식 오류 또는 0원 이하 | 이체 화면에 입력 오류 표시 |
| 잔액 부족 | 이체 거부, 양쪽 잔액과 내역 유지 |
| 출금 후 입금 과정에서 오류 | 출금 내용을 포함해 전체 롤백 |
| 거래내역 저장 중 오류 | 양쪽 잔액과 내역 전체 롤백 |

이체 폼을 다시 보여줄 때는 로그인 회원의 출금 가능 계좌 목록과 기존 입력값을 다시 모델에 담는다.

## 10. 거래내역 조회

```text
GET /accounts/{accountId}/transactions
  -> 로그인 회원 확인
  -> 계좌 존재 여부와 소유권 확인
  -> 계좌 거래내역 최신순 조회
  -> 거래내역 화면
```

- 조회 요청이므로 redirect하지 않는다.
- 없는 계좌나 다른 회원의 계좌이면 거래내역을 노출하지 않는다.

## 11. redirect 목적지 정리

| POST 요청 | 성공 후 이동 |
|---|---|
| `/members/add` | `/login` |
| `/login` | 안전한 `redirectURL` 또는 `/accounts` |
| `/logout` | `/` |
| `/accounts/add` | `/accounts` |
| `/accounts/{accountId}/deposit` | `/accounts/{accountId}` |
| `/accounts/{accountId}/withdraw` | `/accounts/{accountId}` |
| `/transfers/new` | `/accounts/{fromAccountId}` |
