# MiniBank 요구사항 명세서

## 1. 프로젝트 개요

MiniBank는 Spring MVC와 JDBC로 만드는 학습용 미니 인터넷뱅킹 서비스다. 회원가입부터 계좌이체와 거래내역 조회까지 하나의 흐름으로 구현하고, 금융 서비스에서 중요한 **이체 원자성**과 **동시 출금 정합성**을 테스트로 확인하는 것이 핵심 목표다.

v1.0의 완료 상태는 다음과 같이 정의한다.

> 회원가입, 로그인, 계좌 관리와 거래 기능이 실행되고, 데이터가 H2 DB에 저장되며, 이체 롤백과 동시 출금 정합성을 자동화 테스트로 증명할 수 있다.

### 1.1 사용자

v1.0의 사용자는 일반 회원 한 종류다. 회원은 다음 작업을 할 수 있다.

- 회원가입, 로그인, 로그아웃
- 본인 명의의 계좌 개설과 조회
- 본인 계좌의 입금과 출금
- 본인 계좌에서 다른 계좌로 이체
- 본인 계좌의 거래내역 조회

## 2. 범위

## 3. 용어와 데이터 기준

| 용어 | 의미 |
|---|---|
| Member | 로그인 ID로 식별되는 회원 |
| Account | 회원이 소유하며 계좌번호와 현재 잔액을 갖는 계좌 |
| AccountTransaction | 계좌 잔액이 바뀐 사실을 남기는 거래 기록 |
| transferGroupId | 한 번의 이체에서 만들어진 출금·입금 내역을 연결하는 값 |
| 로그인 회원 | 서버 세션에 회원 식별자가 저장된 사용자 |

### 3.1 최소 데이터 항목

| 대상 | 필수 항목 |
|---|---|
| Member | `id`, `loginId`, `password`, `name`, `createdAt` |
| Account | `id`, `memberId`, `accountNumber`, `balance`, `createdAt` |
| AccountTransaction | `id`, `accountId`, `type`, `amount`, `balanceAfter`, `counterpartyAccountId`, `transferGroupId`, `createdAt` |

- 금액은 원 단위 정수로만 다룬다. Java에서는 `long`, DB에서는 `BIGINT`를 사용한다.
- 거래 유형은 입금, 출금, 이체 입금, 이체 출금을 구분할 수 있어야 한다.
- 일반 입금·출금에서는 `counterpartyAccountId`와 `transferGroupId`가 없을 수 있다.

### 3.2 입력값 기준

| 항목 | 기준 |
|---|---|
| 로그인 ID | 영문 소문자와 숫자 4~20자 |
| 비밀번호 | 8~64자, 오류 화면에서 다시 표시하지 않음 |
| 이름 | 공백을 제외한 2~30자 |
| 계좌번호 | 서버가 생성하는 숫자 12자리, 전체 계좌에서 UNIQUE |
| 거래 금액 | 원 단위 `long`, 1원 이상 |

계좌번호는 이체 대상 입력에 사용하므로 v1.0 화면에서 전체 번호를 표시한다. 비밀번호 암호화와 계좌번호 마스킹은 운영 서비스 수준의 보안 범위와 함께 v1.0에서 제외한다.

## 4. 필수 기능

### 4.1 회원가입

- 입력값은 로그인 ID, 비밀번호, 이름이다.
- 세 값은 모두 필수이며 로그인 ID는 중복될 수 없다.
- 가입에 성공하면 로그인 화면으로 이동한다.
- 검증에 실패하면 회원 데이터가 추가되지 않는다.

### 4.2 로그인

- 로그인 ID와 비밀번호가 일치하면 서버 세션에 회원 식별자를 저장한다.
- 보호된 화면에서 로그인한 경우 원래 요청했던 안전한 내부 GET 경로로 돌아간다.
- 복귀 경로가 없거나 유효하지 않으면 계좌 목록으로 이동한다.
- 외부 URL로 redirect하거나 로그인 전에 차단된 POST 요청을 자동 재실행하지 않는다.
- 로그인에 실패하면 세션을 만들지 않고 실패 이유를 보여준다.

### 4.3 로그아웃

- 로그인 회원은 로그아웃할 수 있다.
- 로그아웃하면 현재 로그인 세션을 제거하고 홈으로 이동한다.

### 4.4 계좌 개설

- 로그인 회원은 본인 명의의 계좌를 개설할 수 있다.
- 계좌번호는 시스템이 만들며 전체 계좌에서 중복되지 않는다.
- 계좌번호 충돌 시 최대 5회까지 새 번호로 다시 시도한다.
- 초기 잔액은 항상 0원이다.
- v1.0에서는 계좌 이름이나 초기 잔액을 입력받지 않는다.
- 개설에 성공하면 내 계좌 목록으로 이동한다.

### 4.5 내 계좌 목록 조회

- 로그인 회원이 소유한 계좌만 보여준다.
- 각 계좌의 계좌번호와 현재 잔액을 확인할 수 있다.

### 4.6 계좌 상세 조회

- 본인 계좌의 상세 정보만 조회할 수 있다.
- 계좌번호와 현재 잔액을 보여준다.
- 없는 계좌나 다른 회원의 계좌 정보는 노출하지 않는다.

### 4.7 입금

- 본인 계좌에 1원 이상을 입금할 수 있다.
- 성공하면 잔액이 증가하고 입금 거래내역 1건이 생성된다.
- 잔액 변경과 거래내역 저장은 함께 성공하거나 함께 실패한다.
- 성공 후 해당 계좌 상세로 이동한다.

### 4.8 출금

- 본인 계좌에서 1원 이상을 출금할 수 있다.
- 출금액은 현재 잔액을 넘을 수 없다.
- 성공하면 잔액이 감소하고 출금 거래내역 1건이 생성된다.
- 잔액 부족이나 동시 요청 선처리로 실패하면 잔액과 거래내역이 바뀌지 않는다.
- 성공 후 해당 계좌 상세로 이동한다.

### 4.9 계좌이체

- 본인 계좌에서 존재하는 다른 계좌로 1원 이상을 이체할 수 있다.
- 요청은 출금 계좌의 `fromAccountId`와 입금 계좌의 `toAccountNumber`를 받는다.
- 출금 계좌는 로그인 회원의 소유여야 한다.
- 입금 계좌가 없거나 출금 계좌와 같으면 이체할 수 없다.
- 잔액이 부족하면 이체할 수 없다.
- 출금, 입금, 거래내역 2건 저장은 하나의 트랜잭션으로 처리한다.
- 처리 중 하나라도 실패하면 이체와 관련된 변경을 모두 롤백한다.
- 이체 내역 2건은 같은 `transferGroupId`를 가지며 상대 계좌를 식별할 수 있어야 한다.
- 성공 후 출금 계좌 상세로 이동한다.

Controller는 입금 계좌 ID를 직접 조회하지 않는다. `TransferService`가 계좌번호로 입금 계좌를 찾고 모든 업무 규칙을 검증한다.

```java
transfer(Long loginMemberId,
         Long fromAccountId,
         String toAccountNumber,
         long amount)
```

### 4.10 거래내역 조회

- 본인 계좌의 거래내역만 조회할 수 있다.
- 거래 유형, 금액, 거래 후 잔액, 거래 시각을 보여준다.
- 이체 내역에서는 상대 계좌를 구분할 수 있어야 한다.
- 최신 거래부터 표시하며 수정·삭제 기능은 제공하지 않는다.

## 5. 비즈니스 규칙

### 5.1 금액

- 계좌 잔액은 0원 이상의 정수다.
- 입금·출금·이체 요청 금액은 1원 이상이어야 한다.
- 금액 계산에 부동소수점 자료형을 사용하지 않는다.

### 5.2 잔액과 거래내역

- 계좌 잔액은 음수가 될 수 없다.
- 잔액 변경과 해당 거래내역 저장은 한 작업으로 처리한다.
- 거래내역의 `balanceAfter`는 거래 직후 실제 잔액과 같아야 한다.
- `balanceAfter` 생성 방식과 동시 요청에서의 일관성은 JDBC 구현과 통합 테스트를 거쳐 확정한다.

### 5.3 계좌 소유권

- 회원은 본인 계좌만 조회·입금·출금할 수 있다.
- 이체의 출금 계좌도 본인 소유여야 한다.
- URL이나 화면에 보이는 계좌 ID를 권한으로 간주하지 않고 Service에서 소유권을 확인한다.

### 5.4 이체 원자성

- 출금 계좌 차감, 입금 계좌 증가, 거래내역 2건 저장은 같은 트랜잭션에 포함한다.
- 처리 중 `RuntimeException`이 발생하면 모든 변경을 롤백한다.
- 양방향 동시 이체에서 교착이나 잔액 불일치가 확인되면 일정한 계좌 ID 순서로 잠그는 방식을 검토한다.

### 5.5 거래내역 보존

- 거래내역은 추가만 가능한 append-only 데이터다.
- 생성된 거래내역을 수정하거나 삭제하지 않는다.

### 5.6 동시 출금

- 같은 계좌에 출금 요청이 동시에 들어와도 잔액은 음수가 되지 않는다.
- 출금은 `balance >= amount` 조건을 포함한 단일 `UPDATE`로 처리한다.
- 수정된 행이 1건일 때만 출금 성공으로 판단하고 거래내역을 추가한다.
- 성공한 출금액의 합계, 최종 잔액, 출금 거래내역 수가 서로 맞아야 한다.

### 5.7 웹 요청

- 상태를 바꾸는 요청은 POST를 사용한다.
- 상태 변경에 성공하면 3xx redirect로 GET 화면에 이동해 PRG(Post/Redirect/Get)를 적용한다.
- 로그인 상태는 회원 정보가 아닌 세션 ID 쿠키로 유지한다.

## 6. 예외 처리 기준

실패한 요청은 사용자에게 이유를 알려야 하며, 관련 데이터가 일부만 바뀌어서는 안 된다.

| 상황 | 기대 결과 |
|---|---|
| 로그인 ID 중복 또는 회원가입 필수값 누락 | 회원을 저장하지 않고 입력 오류 표시 |
| 잘못된 로그인 정보 | 세션을 만들지 않고 로그인 오류 표시 |
| 비로그인 상태로 보호 기능 접근 | 로그인 화면으로 이동 |
| 존재하지 않는 계좌 | 요청 거부, 계좌 정보 비노출 |
| 다른 회원의 계좌 접근 | 요청 거부, 데이터 변경 없음 |
| 0원 이하 거래 금액 | 요청 거부, 데이터 변경 없음 |
| 잔액보다 큰 출금·이체 | 요청 거부, 잔액과 내역 유지 |
| 동일 계좌 이체 | 요청 거부, 데이터 변경 없음 |
| 이체 처리 중 시스템 오류 | 잔액과 내역 전체 롤백 |
| 동시 출금 중 잔액 조건 불충족 | 해당 요청만 실패, 성공한 거래와 최종 잔액 유지 |

## 7. 품질과 구현 기준

### 7.1 데이터 정합성

- 계좌의 현재 잔액과 성공한 거래내역은 항상 일치해야 한다.
- 이체 실패 시 일부 잔액이나 일부 거래내역만 남아서는 안 된다.
- DB의 `accounts.balance`는 0 이상이어야 한다.

### 7.2 보안과 접근 제어

- 계좌와 이체 기능은 로그인 회원만 사용할 수 있다.
- 인증은 세션과 인터셉터로 확인한다.
- 계좌별 권한은 Service에서 로그인 회원 ID와 계좌 소유자 ID를 비교해 확인한다.
- 로그인 후 복귀 경로는 애플리케이션 내부의 안전한 GET 경로만 허용한다.
- Spring Security와 BCrypt는 v1.0 범위에서 제외한다. 따라서 실제 금융 서비스 수준의 보안 구현으로 간주하지 않는다.

### 7.3 자동화 테스트

최소한 다음 항목을 자동화 테스트로 확인한다.

- 회원가입 성공과 로그인 ID 중복 실패
- 계좌 개설과 회원별 계좌 조회
- 타인 계좌 접근 실패
- 0원·음수 거래 실패
- 정상 입금과 정상 출금
- 잔액 부족 출금 실패
- 정상 계좌이체
- 이체 중간 실패 시 전체 롤백
- 동일 계좌 동시 출금 시 잔액 정합성

### 7.4 구조

- 요청 처리 흐름은 `Controller -> Service -> Repository`를 따른다.
- Controller는 Repository를 직접 호출하지 않는다.
- Repository는 인터페이스와 Memory/JDBC 구현체로 나눈다.
- 웹 흐름을 바꾸지 않고 저장소 구현체를 교체할 수 있어야 한다.

### 7.5 실행과 문서

- `./gradlew clean test`로 전체 테스트를 실행할 수 있어야 한다.
- `./gradlew bootRun`으로 로컬에서 실행할 수 있어야 한다.
- H2와 `schema.sql`을 사용한다.
- README에 실행 방법, ERD, 트랜잭션 경계, 롤백 검증, 동시 출금 문제와 해결 과정을 기록한다.

## 8. 핵심 테스트 시나리오

### 8.1 정상 이체

```text
Given A 계좌 잔액은 100,000원, B 계좌 잔액은 0원이다.
When  A 계좌에서 B 계좌로 20,000원을 이체한다.
Then  A 잔액은 80,000원, B 잔액은 20,000원이다.
And   양쪽 거래내역이 1건씩 생성되고 transferGroupId가 같다.
```

### 8.2 이체 롤백

```text
Given A 계좌 잔액은 100,000원, B 계좌 잔액은 0원이다.
When  A 계좌 차감 후 B 계좌 증가 전에 RuntimeException이 발생한다.
Then  A 잔액은 100,000원, B 잔액은 0원이다.
And   해당 이체의 거래내역은 남지 않는다.
```

### 8.3 동시 출금

```text
Given 계좌 잔액은 10,000원이다.
When  두 스레드가 동시에 7,000원씩 출금을 요청한다.
Then  성공 1건, 실패 1건이다.
And   최종 잔액은 3,000원이며 출금 거래내역은 1건이다.
```
